# Context Budget CN

用在 Codex 对话、skill、代码审查、仓库分析或文档任务开始变长时。目标不是“少说话”，而是把上下文预算花在真正影响结果的证据、约束和文件上。

## 适用场景

- 用户说“省 token”“上下文太长”“帮我压缩 prompt”“这个 skill 太占上下文”。
- 需要把长对话、长 issue、长 PR、长日志压缩成高信号摘要。
- 需要设计 Codex skill 的渐进式加载，避免 `SKILL.md` 过大。
- 需要在大仓库里控制读取文件范围。

## 不适用场景

- 任务本身很短，不需要额外压缩。
- 压缩会丢失安全、法律、财务、医疗等高风险细节。
- 用户要求删除关键约束、失败日志或验证证据来“省 token”。

## 工作流

1. 明确任务目标。
   - 先写一句话目标：要解决什么问题？
   - 只保留服务目标的信息。

2. 分层上下文。
   - 必需：用户目标、约束、相关文件、失败日志、验证结果。
   - 有用：设计偏好、历史决策、相关 issue。
   - 可丢：重复解释、长篇背景、无关文件、营销文字。

3. 控制读取范围。
   - 先用 `rg --files`、`rg`、目录树和 README 建立地图。
   - 只读与任务相关的文件。
   - 大文件先搜索关键符号或章节，再局部读取。

4. 做高信号摘要。
   - 保留事实、路径、行号、错误、决策、开放问题。
   - 删除重复措辞和无关叙事。
   - 对推断明确标注“推断”。

5. 设计渐进式 skill。
   - `SKILL.md` 只放触发条件、核心工作流、边界和验证。
   - 详细例子、长规范、模板放到 `references/` 或 `examples/`。
   - 在 `SKILL.md` 中说明什么时候读取额外文件。

6. 验证压缩质量。
   - 压缩后检查是否还能执行任务。
   - 确认没有丢失安全边界、失败条件、验收标准和关键路径。

## 输出格式

```text
任务目标：

必须保留：
- ...

可以删除/延后：
- ...

压缩摘要：
- ...

后续需要时再读取：
- ...

风险：
- ...
```

## 验证方式

- 压缩后应能回答：目标是什么、限制是什么、下一步做什么、如何验证。
- 对代码任务，至少保留相关文件路径和关键符号。
- 对 bug/CI 任务，至少保留错误信息、复现路径和已运行命令。

## 安全边界

- 不要为了省 token 删除用户明确约束。
- 不要删除安全、隐私、权限、合规相关信息。
- 不要把不确定事实压缩成确定结论。
