# Codex 扩展边界：skill、MCP、plugin 与个性化设置

这份说明回答 issue #4 里的核心问题：什么时候应该写 skill，什么时候应该接 MCP，什么时候才需要 plugin，以及哪些内容只适合放进个人 Codex 设置。

## 快速判断

| 需求 | 优先选择 | 原因 |
| --- | --- | --- |
| 固化一套工作流、判断标准、输出格式或安全边界 | skill | skill 是给 Codex 的任务说明，适合沉淀可复用的协作方法 |
| 读取数据库、调用内部 API、访问本地服务或暴露结构化工具 | MCP | MCP 适合把外部能力变成 Codex 可调用的工具 |
| 打包一组 skills、MCP、应用能力或团队分发配置 | plugin | plugin 适合发布和组合多种 Codex 扩展能力 |
| 个人偏好的语气、默认原则、常用路径或长期约束 | 个性化设置 / AGENTS.md | 这些内容是长期偏好，不应该伪装成某个特定任务 skill |

## 什么时候用 skill

用 skill 的前提是：Codex 已经能通过现有工具完成任务，但需要更稳定的流程、边界和输出标准。

适合写成 skill 的例子：

- PR review：要求先看 diff 和测试，再按严重程度输出问题。
- issue triage：把模糊反馈转成复现信息、优先级和维护者回复。
- release notes：从 commits、PR 或 changelog 草稿整理用户能读懂的发布说明。
- security advisory triage：规定哪些信息不能公开，如何区分 confirmed、likely、unknown。

不适合写成 skill 的例子：

- 需要一个新工具去读取私有系统。
- 需要长期运行的服务。
- 需要把账号、密钥或浏览器会话暴露给 Codex。
- 只是某个用户的固定写作口吻。

## 什么时候用 MCP

当问题的关键不是“Codex 应该怎么思考和汇报”，而是“Codex 缺少一个可调用能力”时，优先考虑 MCP。

适合 MCP 的例子：

- 查询公司内部 issue 系统。
- 读取数据库 schema 或运行受控查询。
- 调用日志、监控、搜索、爬取、截图等工具。
- 提供严格参数、结构化返回值和权限边界。

MCP 不应该替代 skill。一个 MCP 只提供工具能力时，仍然需要 skill 或项目说明来规定：

- 什么时候调用这个工具；
- 调用前要不要先征得用户同意；
- 如何处理失败、超时、敏感数据；
- 输出给用户时应该保留哪些证据。

## 什么时候用 plugin

plugin 更像分发单元，而不是单个工作流。只有在需要把多种能力一起交付时才考虑 plugin。

适合 plugin 的例子：

- 一个团队想分发一组专用 skills 和 MCP servers。
- 一个产品想提供浏览器、GitHub、Figma 等配套能力。
- 一个能力需要 manifest、资产、脚本和说明一起安装。

如果只有一份 `SKILL.md`，不要为了显得正式而做成 plugin。

## 什么时候用个性化设置或 AGENTS.md

长期、全局、偏个人或偏项目约束，适合放在个性化设置或仓库级 `AGENTS.md`，不适合写成通用 skill。

适合放这里的内容：

- “先读代码再改。”
- “默认运行最相关的测试。”
- “不要重构无关文件。”
- “这个仓库的验证命令是 `python scripts/validate-skills.py`。”

如果这些规则只在某一类任务里有意义，再考虑把它们整理进对应 skill。

## 组合方式

实际项目里经常组合使用：

1. `AGENTS.md` 记录仓库长期约束。
2. skill 记录某类任务的工作流和质量标准。
3. MCP 提供任务需要的外部工具。
4. plugin 把一组 skills 和 MCP 打包分发。

一个好的边界判断是：先问“缺的是流程，还是工具？”

- 缺流程：写 skill。
- 缺工具：接 MCP。
- 缺打包分发：做 plugin。
- 缺长期偏好：写进个性化设置或 `AGENTS.md`。
